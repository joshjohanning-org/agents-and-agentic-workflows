# Setup Guide

This guide walks you through setting up GitHub Agentic Workflows in your repository.

## Prerequisites

Before you begin, ensure you have:

- **GitHub CLI** (`gh`) v2.0.0+ -- [Install here](https://cli.github.com/)
- **GitHub Repository** with write access and GitHub Actions enabled
- **AI Account** -- one of:
  - [GitHub Copilot](https://github.com/features/copilot) (recommended)
  - [Anthropic Claude](https://www.anthropic.com/)
  - [OpenAI Codex](https://openai.com/api/)

## Step 1: Install the CLI Extension

```bash
gh extension install github/gh-aw
```

If you encounter authentication issues:

```bash
# Option A: Use the install script
curl -sL https://raw.githubusercontent.com/github/gh-aw/main/install-gh-aw.sh | bash

# Option B: Login interactively first
gh auth login
gh extension install github/gh-aw
```

## Step 2: Add a Workflow

### Option A: Use the Interactive Wizard (Recommended)

```bash
gh aw add-wizard githubnext/agentics/daily-repo-status
```

The wizard will:
1. Verify repository permissions
2. Let you select an AI engine (Copilot, Claude, or Codex)
3. Set up the required secret (`COPILOT_GITHUB_TOKEN`, `ANTHROPIC_API_KEY`, or `OPENAI_API_KEY`)
4. Add the workflow and lock file to `.github/workflows/`
5. Optionally trigger an initial run

### Option B: Add Workflows Manually

1. Copy a workflow `.md` file into `.github/workflows/` in your repository
2. Compile the lock file:

```bash
gh aw compile
```

3. Commit and push both files (the `.md` and the generated `.lock.yml`)

### Option C: Create Your Own Workflow with AI Assistance

Use this prompt with your favorite coding agent:

> Generate a workflow that creates a daily repo status report for a maintainer. Use the instructions at https://github.com/github/gh-aw/blob/main/create.md

The agent will guide you through creating both the `.md` workflow file and the `.lock.yml` lock file.

## Step 3: Configure Secrets

Depending on your chosen AI engine, add the appropriate secret to your repository:

| Engine | Secret Name | How to Get It |
|---|---|---|
| **Copilot** | `COPILOT_GITHUB_TOKEN` | [Authentication docs](https://github.github.com/gh-aw/reference/auth/#copilot_github_token) |
| **Claude** | `ANTHROPIC_API_KEY` | [Anthropic Console](https://console.anthropic.com/) |
| **Codex** | `OPENAI_API_KEY` | [OpenAI Platform](https://platform.openai.com/api-keys) |

Add the secret in your repository: **Settings > Secrets and variables > Actions > New repository secret**

## Step 4: Trigger a Run

```bash
# Run a specific workflow
gh aw run daily-repo-status

# Or wait for the configured trigger (schedule, issue opened, etc.)
```

Workflow runs typically take 2-3 minutes to complete.

## Step 5: Customize

1. Open the workflow `.md` file in `.github/workflows/`
2. Edit the natural language instructions to match your needs
3. If you changed frontmatter settings, recompile:

```bash
gh aw compile
```

4. Commit and push the updated files

## Workflow Files Explained

Each agentic workflow consists of two files:

- **`workflow-name.md`** -- The editable source of truth. Contains YAML frontmatter for configuration and Markdown for instructions.
- **`workflow-name.lock.yml`** -- The compiled GitHub Actions workflow. Generated by `gh aw compile`. Do not edit manually.

Both files must be committed to `.github/workflows/`.

## Understanding Frontmatter

The frontmatter section (between `---` markers) configures:

```yaml
---
# When the workflow runs
on:
  issues:
    types: [opened]

# What the agent can access (read-only by default)
permissions:
  issues: read
  contents: read

# Which tools the agent can use
tools:
  github:
    toolsets: [issues, labels]

# What write operations the agent can perform (safe outputs)
safe-outputs:
  add-comment: {}
  add-labels:
    allowed: [bug, feature, enhancement]
---
```

## Cost Considerations

- Each workflow run typically incurs **2 premium requests** when using Copilot:
  one for the agentic work and one for a guardrail check through safe outputs
- You can configure different AI models to help manage costs
- Start with low-frequency schedules (weekly) and increase as needed
- Use `workflow_dispatch` triggers for manual testing before enabling schedules
